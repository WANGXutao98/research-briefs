### Question 1: Does OpenEnv's Standardized Interface Make WMATS-Bench's Evaluation Protocol Trivially Reproducible?

Every piece of WMATS-Bench's evaluation pipeline maps directly onto OpenEnv's abstractions: world model as an environment wrapper, rollout generation as an environment step, Diffusion Policy as the RL agent. The question: can WMATS-Bench be built as an OpenEnv plugin (a collection of `WorldModelEnv` wrappers, one per WAM architecture), making it trivially reproducible by anyone in the HuggingFace ecosystem without local simulation infrastructure?

**Why It Matters Now**: OpenEnv is brand new (trending this week, ★2,268 already — fast growth signal). **Why It's Tractable**: OpenEnv's Python API is designed exactly for this use case. Wrapping AHEAD as an `OpenEnv`-compatible environment (the anchor experiment's first artifact) requires: subclassing OpenEnv's `Environment` base class, implementing `step()` to call AHEAD's forward pass, and implementing `reset()` to initialize MuJoCo peg insertion. Estimated: 2-3 days of integration work on top of the anchor experiment.

**Evidence**: `huggingface/OpenEnv` (GitHub, ★2,268, trending week of 2026-06-15), AHEAD (arXiv 2606.02486), WMATS-Bench design from 2026-06-14_Ideas.md.

### Question 2: Can the "Dreaming" Memory Consolidation Pattern Solve World Model Drift in Long-Horizon Tasks?

**The Question**: OpenAI's "Dreaming" (ChatGPT, June 4, 2026) is an offline consolidation process that re-processes prior conversations to refresh and refine stored user preferences — explicitly named after biological sleep-consolidation theories of memory. For world models, the analogous problem is **temporal drift**: after 30+ seconds of autoregressive rollout, world models systematically drift from physical ground truth because small prediction errors compound. The question: can a "dreaming" step — offline replay of collected trajectory data to fine-tune a world model's transition function between deployment episodes — reduce long-horizon drift without catastrophic forgetting of previously learned dynamics?

**Why It Matters Now**: OpenAI's deployment of offline consolidation at scale confirms that the memory replay approach (well-known in continual RL as Dreamer/DreamerV3 offline phases) is converging with production systems. For embodied AI, this is timely: the WorldOlympiad benchmark (arXiv 2606.11129) already identifies long-horizon physical consistency as a gap. Cosmos Predict 2.5 generates only 30-second videos. The 30-second frontier is now well-documented, making anything beyond it high-opportunity.

**Why It's Tractable**: DreamerV3's offline replay mechanism is open-source and well-understood. The adaptation required: after each real-robot episode, run a "dream" pass — fine-tune the world model's transition network on the collected trajectory using LoRA (low-rank adaptation, ≤10M params) for 100 gradient steps. Evaluate: does drift (measured as prediction MSE at step 60s vs. step 30s) decrease after dreaming? Use the Physical AI Dataset force/torque trajectories as the replay buffer. H100x1, 1 week.

**Evidence**: OpenAI "Dreaming" (ChatGPT, June 4, 2026), WorldOlympiad (arXiv 2606.11129), Cosmos Predict 2.5 (30s frontier), NVIDIA Physical AI Dataset (open, 500K trajectories, 4.8M+ downloads), DreamerV3 (Hafner et al., ICLR 2024).

### Question 4: Does Encoder-Free Multimodal Architecture (Gemma 4 12B) Change How World Models Should Process Sensor Streams?

**The Question**: Google's Gemma 4 12B (Apache 2.0, runs on 16GB laptop) processes text, images, video, and audio natively within a decoder architecture — without separate vision/audio encoders. Every current video world model (Cosmos 3, AHEAD, BiWM) uses separate visual encoders (ViT, CLIP, or similar) that must be pretrained and aligned separately. The question: does an encoder-free architecture for world model perception — processing raw robot sensor streams natively within the world model's decoder — produce better or worse latent representations for physical dynamics prediction, compared to the dominant encoder-then-decode pipeline?

**Why It Matters Now**: Gemma 4 12B's encoder-free design at 12B parameters (Apache 2.0, deployable locally) makes this question empirically testable for the first time without training from scratch. Cosmos 3's architecture is a two-tower MoT that still separates perception from generation. If encoder-free world model perception works better for multi-modal robot inputs (RGB + force/torque + proprioception), it eliminates the modality-specific pretraining bottleneck that currently limits novel sensor modality integration.

**Why It's Tractable**: Fine-tune Gemma 4 12B on a robot manipulation prediction task — given 5 frames of RGB + joint state, predict next 5 frames + next joint state. Compare against a ViT-encoder + decoder baseline of equivalent parameter count on the same task. Use the Physical AI Dataset. H100x8, 2 weeks for fine-tuning. H100x1 for inference comparison. If encoder-free matches encoder-based on physics prediction, the finding is architecturally significant.

**Evidence**: Gemma 4 12B (Google DeepMind, June 4, 2026), Cosmos 3 (arXiv 2606.02800, two-tower MoT), NVIDIA Physical AI Dataset (open, 500K trajectories).

### Idea 3: Hy-WAM — Compact World-Action Models for Real-Time Closed-Loop Control

- **One-liner**: Building on Hy-Embodied-0.5-VLA's open deployment stack, a systematic study of whether compact (≤1B parameter) world-action models can match 7B+ WAMs on task success rate when deployed at native inference frequency (10–20 Hz) rather than in the offline-rollout regime most WAMs assume.
- **Core Hypothesis**: Large WAMs (Cosmos 3, AHA-WAM) are evaluated offline — their rollouts are pre-computed and used as offline training data. In closed-loop deployment, latency (inference time per step) fundamentally limits task success on dynamic tasks. A 0.5B WAM running at 20 Hz outperforms a 7B WAM running at 1 Hz on tasks requiring reactive correction (moving objects, grasp slip recovery), even if the 7B model has lower offline prediction MSE.
- **Novelty vs Existing Work**:
  - Efficient-WAM (arXiv 2606.10040): targets inference speed but remains a 1B-class model, not compared against frequency-constrained 7B alternatives.

  - Hy-Embodied-0.5-VLA (GitHub June 12): provides the deployment stack but no world model component.
  - **What's new**: First paper to evaluate WAM architectures under a **latency budget constraint** (must run at ≥10 Hz for closed-loop control), identifying the Pareto frontier between model capacity and deployment frequency for dynamic manipulation tasks.
